# Monthoya refactor roadmap

This document is the working plan for reducing large-file edits, lowering AI/Codex token usage, and making future features safer to add.

## Why this is needed

The app has grown past the first prototype stage. Several files now contain many unrelated responsibilities. This makes small changes expensive and risky because a change to one field can touch a large XAML file, large code-behind file, entities, services, migrations, and page state.

The recent IPTU split is a good example: a small business change became fragile because the form layout, save logic, default page state, request object, entity, and migration were tightly coupled.

## Refactor rules

1. Prefer small, mechanical refactors before behavior changes.
2. Each refactor should build before the next one starts.
3. Avoid giant positional constructors for UI state.
4. New screens should not be added directly into ShellWindow.xaml.
5. New module logic should not be added directly into ShellWindow.* unless it is temporary.
6. Keep migrations generated by EF whenever possible.
7. Remove temporary repair scripts after the related change is stable.

Monthoya.Desktop/
  Views/
    ShellWindow.xaml
    ShellWindow.xaml.cs
    Imoveis/

This is a target, not a one-commit rewrite.

## Phase 1: audit and cleanup

- Add scripts to identify large files and temporary repair scripts.
- Confirm the solution builds.
- Delete temporary one-off repair scripts after the current IPTU change is stable.
- Commit generated EF migration and package changes cleanly.

## Phase 2: Imoveis module split

Start with Imoveis because it currently has high churn and many fields.

Suggested order:

1. Move `ImoveisPageState` out of `ShellWindow.Imoveis.cs` into a dedicated file.
2. Convert `ImoveisPageState` from a positional record into a property-based class or smaller state objects.
3. Extract the Imoveis form section from `ShellWindow.xaml` into `Views/Imoveis/ImovelFormView.xaml`.
4. Extract media/fotos into `Views/Imoveis/ImovelMediaView.xaml`.
5. Extract keys/chaves into `Views/Imoveis/ImovelKeysView.xaml`.
6. Leave ShellWindow responsible only for navigation and page hosting.

## Phase 3: repeat for other modules

After Imoveis is stable, repeat the same pattern for:

- Pessoas
- Locações
- Financeiro
- Boletos
- Notas Fiscais
- Documentos
- Relatórios
- Configurações

## Phase 4: safety automation

Add a simple GitHub Actions workflow:

```text
dotnet restore
dotnet build Monthoya.sln
dotnet test Monthoya.Tests
```

This should run on every push and pull request.

## Phase 5: database/design-time cleanup

The current design-time EF connection can fall back to localhost/monthoya_design_time. That is useful only when local PostgreSQL is running. Long term, design-time EF should read a local developer connection string from user secrets or environment variables.

## Done criteria for each refactor

A refactor is complete only when:

- `dotnet build Monthoya.sln` succeeds;
- `dotnet test Monthoya.Tests` succeeds;
- no one-off repair script is needed for normal development;
- the changed module is easier to edit than before;
- a future field change touches fewer files.
